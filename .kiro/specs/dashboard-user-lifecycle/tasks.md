# Implementation Plan — dashboard-user-lifecycle

> 発生源: GitHub Issues #31（無効化の不可逆性）・#32（無効化ガードの欠如）。
> 依存する設計: `design.md`（advisory lock による並行ガード・破壊的変更のノーレッドベースライン配置・API 契約表）。
> 共有 DB テストの UUID 接頭辞は **f8**（f7 まで使用済み）。

- [ ] 1. DAL 基盤: 利用者ライフサイクルのアクセサ（@fwlm/db）
- [x] 1.1 再有効化とスコープ限定メールルックアップのアクセサを追加
  - 無効化済み利用者を有効に戻すアクセサ（無効化時刻を解除）。既に有効でも同じ内容を返し冪等に振る舞う。
  - 対象が呼び出し運営の管理範囲外・不在なら「該当なし」を返す（呼び出し側で 404 に写像する前提）。
  - 同一メール（大文字小文字無視）が呼び出し運営の配下に存在するかを、無効化状態つきで引くスコープ限定ルックアップ。他運営配下は「該当なし」を返し越境を秘匿する。
  - 追加のみで既存挙動を壊さない（初回ログイン紐付けの条件は変更しない）。
  - 完了状態: 実 DB テスト（新規フィクスチャ接頭辞 f8）で、リンク済み無効行の有効化で再ログイン可能な状態に戻ること、保留無効行の有効化で初回ログイン紐付けが再開できること、冪等性、範囲外 null、越境ルックアップの秘匿が緑になる。
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 3.2, 4.1_

- [x] 1.2 最後の運営を保護する無効化アクセサ（テナント単位の直列化）を追加
  - トランザクション内でテナント（運営）単位の advisory ロックを取得して同一テナントの無効化を直列化し、対象を無効化すると有効な運営（保留＝未ログイン運営を含む）が0人になる場合のみ拒否を表す結果を返す。
  - 無効化成功／既に無効（冪等）／最後の運営で拒否／対象なし、を判別可能な結果として返す。呼び出し運営の範囲外・不在は「対象なし」。
  - トランザクション接続を受け取るための最小面をプール層に追加する（既存の接続可能プールの前例に合わせる）。
  - この段階では既存の単純な無効化アクセサは残置し、追加のみでビルド緑を保つ（切替と旧アクセサ撤去は 2.1 で行う）。
  - 完了状態: 実 DB テスト（f8・非並行シナリオ）で、有効運営2名以上なら運営を無効化でき、有効運営1名のみならその運営は拒否され、代理店は常に無効化でき、対象なしが判別されることが緑になる。
  - _Requirements: 2.3, 2.4, 4.1_

- [x] 1.3 無効化ガードの並行安全性を決定的に検証
  - 2 つの DB クライアント接続を割り込み制御し、最後の2運営を同時に無効化する競合を再現する。先行がロックを保持したまま片方を無効化・commit した後、後続がブロック解除で最新状態を観測し、必ず一方が「最後の運営」で拒否されることを確認する。
  - 完了状態: 並行テスト（f8）が決定的に緑で、いかなる割り込み順序でも有効運営が最低1名残ることが実証される。
  - _Requirements: 2.5_
  - _Depends: 1.2_

- [ ] 2. dashboard-api: 無効化ガード・再有効化・登録案内・自己識別
- [x] 2.1 無効化ハンドラを保護付きへ切替（自己無効化拒否・結果の HTTP 写像・全配線同時更新）
  - 認可（運営限定）通過後、DB 到達前に「対象が自分自身」なら専用の 409 で拒否する。
  - 保護付き無効化アクセサへ切り替え、結果（成功／最後の運営拒否／対象なし）を 200／409／404 に写像する。拒否時は対象の状態を変えない。
  - 旧・単純無効化アクセサを撤去し、それを import・配線している dashboard-api の全テスト（利用者一覧の配線検証・招待/リンク統合・店舗登録統合の各実 DB テスト）と、そのアクセサ自身の DAL テストを、保護付き版へ同時に差し替える。無効化系依存の合成根配線も保護付き版へ更新する。
  - 完了状態: 単体テストで自己無効化 409・最後の運営 409・通常の運営/代理店無効化 200・対象なし 404 になり、旧アクセサ参照が全ファイルから消えて dashboard-api パッケージ全体がビルド・テスト緑。
  - _Requirements: 2.1, 2.3, 2.4, 2.6, 1.5_
  - _Depends: 1.2_

- [x] 2.2 再有効化のエンドポイント・ハンドラ・配線を追加（全 AppDeps 構築の同時更新）
  - 運営限定の再有効化エンドポイントとハンドラを追加。対象なし・範囲外は 404、成功は対象を返す 200。合成根で再有効化の実依存を配線する。
  - 再有効化依存が必須化されるため、完全な依存一式を構築する全箇所（合成根・アプリ結線テスト・招待/リンク統合・店舗登録統合の各テスト）に再有効化依存を追加してビルドを緑に保つ。
  - 完了状態: 配線／実起動系テストで再有効化エンドポイントが運営のみ 200・非運営 403 になり、dashboard-api パッケージ全体がビルド緑。

## Implementation Notes
<!-- kiro-impl が横断的知見を追記する。既知の前提: 破壊的変更（旧無効化アクセサ撤去・無効化結果型変更・/me への id 追加）は該当テストと同一タスクで更新しビルド緑を保つこと（design「実装順序の制約」）。共有 DB テストの UUID 接頭辞は f8 から。 -->
- 1.2: advisory lock は `pg_advisory_xact_lock($1::int4, hashtext($2)::int4)`（第1引数=固定クラス `DISABLE_LOCK_CLASS=0x64756c31`・第2引数=hashtext(operator_id)）でテナント直列化。`TransactionCapable`=`Pick<Pool,'connect'>` 相当を pool.ts に追加（pg Pool が構造適合）。guard フィクスチャは f8 別帯（op 9001/9002・user e0000000XXXX）で 1.1 の f8（a1/b2・d0000000000X）と非交差。
- 1.3: **並行テストは Promise.all のタイミング依存では非決定的**（環境で ~95% 自然直列化し lock 除去でも通ってしまう）。正解は**別接続でテナント advisory ロックを保持し op2 を未コミット無効化→本物ガードがブロック（500ms 有界待機で settled=false）→holder COMMIT→ガードが last_operator を返す**という TX 制御。lock 除去で 100% 失敗する（決定的・突然変異検証済み）。並行フィクスチャは f8 帯 op 9003・user e1000000XXXX。
- レビュー規律: subagent の突然変異検証は **Edit 逆置換で復元し `git checkout` 禁止**（未コミット差分破棄の事故防止）。コミット前に親が `git diff --name-only`＋テスト再実行で独立確認する。
- 実行環境: session limit・接続断で implementer subagent が中断する場合あり。その際は親のメインコンテキストで manual mode 実装/remediation＋独立レビュー subagent に切替（1.2/1.3/2.2 で実施）。
- 2.2: enable の DoD「配線／実起動系テストで運営のみ200・非運営403」は app-routes.db.test に enable ルートの app.request アサート（no-auth 401・agency 403・operator 200＋DB disabled_at NULL）で満たす。ルート名破壊で当該テストが落ちることを確認（ハンドラ単体だけでは配線リグレッションを検出できない）。
- 2.3→4.1 引き継ぎ: index.ts の DI 引数順転置（findUserByEmailInOperator の email↔operatorId）は 23505 経路の実配線 DB テストが無く突然変異が生存（未検出）。→ **4.1 で解消**: app-routes.db.test に「自運営無効化済みメール衝突→email_conflict_disabled」「越境衝突→email_conflict」を実 DI で追加（buildApp 複製配線の転置を 500/コード化で決定的捕捉）。
- 4.1 の構造的制約（将来強化候補・非ブロッキング）: app-routes.db.test は `index.ts` を直接 import できない（`serve()`/`initializeApp()` のモジュールロード副作用）ため、DI を **buildApp で複製**して検証する。この複製は index.ts とバイト同一を慣例で維持するが、**index.ts 単独の乖離（複製と非同期な転置）は捕捉できない**。恒久対策は DI 構築を import 可能なファクトリへ抽出して index.ts とテストで共有する案（別 spec 候補）。全ルート配線テストに共通する既存パターンの制約であり本 feature 固有ではない。
