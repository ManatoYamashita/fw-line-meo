# Requirements Document

## Project Description (Input)

**誰が課題を持つか**: 飲食店オーナー（LINEだけで一元管理したい・ITに不慣れ）と、運営（機能1・機能3を実オーナーに届けたい）。

**現状**: 機能3（review-acquisition）は完成済みだが、オーナーが実際に LINE 友だち追加からサービスに参加する経路が存在しない。`owners`／`stores` テーブルは `onboarding_status`（`pending`→`store_identified`→`active`）・`place_status`（`pending`/`confirmed`）を備えているが、これらを遷移させる LINE Webhook 基盤・会話フローが未実装。全 LINE 系機能（機能1 の Flex 配信・将来の機能2）が乗る Webhook 基盤自体も存在しない。

**何を変えるべきか**: LINE Webhook 基盤（署名検証・イベント dispatch・重複排除）を構築し、オーナーが「友だち追加 → 代理店の招待コード入力 → 店名検索 → Google Places 候補から選択・確認」という LINE 内チャット完結フローで店舗特定（Place ID 確定）まで到達できるようにする。完了時、owner の `onboarding_status` は `store_identified` となり、機能1（競合日次サマリー配信）の対象として登録可能な状態になる。

詳細な背景・アプローチ・スコープ境界は discovery 済みの `.kiro/specs/line-onboarding/brief.md` を参照。

## Introduction

本仕様は、飲食店オーナーが運営保有の単一 LINE 公式アカウントを友だち追加してから、自店舗を Google 上の店舗（Place）として特定し終えるまでの「段階的オンボーディング」を、LINE のトーク内で完結する体験として定義する。オーナーは代理店ごとに発行された招待コードで正しい代理店に紐付き、店名を送るだけで候補から自店を選んで確定できる。完了したオーナーは「店舗特定済み」となり、機能1（競合日次サマリー）の配信対象として登録可能な状態に到達する。

以下の受け入れ基準における主語「オンボーディングサービス」は、本仕様が定義する LINE 上のオーナー向け受付・案内機能全体を指す。

## Boundary Context

- **In scope**: 友だち追加から店舗特定までの LINE トーク内完結フロー／招待コードの検証と代理店紐付け／店名検索による店舗候補の提示と確定／オーナーごとの進捗状態の保持・再開／常設メニュー（リッチメニュー）の最小形
- **Out of scope**: 招待コードの発行 UI（コードは運営が別手段で事前発行。UI 化は代理店ダッシュボード＝Issue #5）／代理店ダッシュボードでの店舗登録代行（Issue #5）／機能1 の配信そのもの・配信時刻等のオーナー設定 UI（Issue #4）／Google OAuth・機能2（第2フェーズ）／店舗特定済み確定後の店舗変更（運営対応）／客向け機能（機能3）への変更
- **Adjacent expectations**: 機能1（Issue #4）は「店舗特定済み」に到達したオーナーを配信対象として扱う想定である。代理店ダッシュボード（Issue #5）は本機能と同等の店舗特定能力を代行経路として利用する想定である。招待コードは本機能の利用開始前に運営によって発行済みであることを前提とする。

## Requirements

### Requirement 1: 友だち追加と受付開始

**Objective:** As a 飲食店オーナー, I want 公式アカウントを友だち追加するだけでオンボーディングが始まる体験, so that 特別な準備や説明書なしにサービスの利用を開始できる

#### Acceptance Criteria

1. When 未登録の LINE ユーザーが公式アカウントを友だち追加したとき, the オンボーディングサービス shall 挨拶と招待コードの入力案内をトークで送信する
2. When 登録済みのオーナーがブロック解除等により再び友だち追加したとき, the オンボーディングサービス shall 保存済みの進捗に応じた次の手順を案内し、オーナー情報を重複して作成しない
3. While オーナーが招待コード入力前の状態, the オンボーディングサービス shall 店舗特定に関する操作を受け付けない

### Requirement 2: 招待コードによる代理店への紐付け

**Objective:** As a 運営・代理店, I want オーナーが招待コード経由で正しい代理店に紐付くこと, so that 運営→代理店→オーナーの管理構造を崩さずにオーナーを受け入れられる

#### Acceptance Criteria

1. When 招待コード入力待ちのオーナーが有効な招待コードを送信したとき, the オンボーディングサービス shall 当該コードを発行した代理店に紐付くオーナーとして登録し、店名の入力を案内する
2. If 無効な招待コードが送信されたとき, the オンボーディングサービス shall オーナーを登録せず、コードが正しくない旨と再入力の案内を送信する
3. If 同一の LINE ユーザーから無効な招待コードが連続 5 回送信されたとき, the オンボーディングサービス shall 一定時間コード入力の受付を停止し、その旨を案内する
4. The オンボーディングサービス shall 代理店への紐付けが確定していないオーナーを登録しない
5. The オンボーディングサービス shall 招待コードを代理店ごとに区別し、コードが無効化されない限り同一コードで複数のオーナーを登録できるものとする

### Requirement 3: 店名検索による店舗候補の提示

**Objective:** As a 飲食店オーナー, I want 店名を送るだけで自店の候補が提示されること, so that IT 操作に不慣れでも自店舗を特定できる

#### Acceptance Criteria

1. When 店名入力待ちのオーナーが店名のテキストを送信したとき, the オンボーディングサービス shall Google 上の店舗情報から候補を検索し、店名と住所を含む候補一覧（最大 10 件）を選択可能な形式で提示する
2. If 検索結果が 0 件のとき, the オンボーディングサービス shall 見つからなかった旨と、表記を変えた再入力の案内を送信する
3. If 店舗候補の検索が外部要因で失敗したとき, the オンボーディングサービス shall エラーである旨と再試行の案内を送信し、オーナーの進捗状態を失わない
4. When 候補一覧の提示後にオーナーが別の店名テキストを送信したとき, the オンボーディングサービス shall 新しい店名で再検索し候補を提示し直す
5. The オンボーディングサービス shall 店舗候補の取得に Google が公式に提供する手段のみを用いる（Web 画面の自動収集等は行わない）

### Requirement 4: 店舗の確定と「店舗特定済み」への遷移

**Objective:** As a 飲食店オーナー, I want 候補から自店を選んで確定すること, so that 機能1（競合日次サマリー）の配信対象として登録される状態に到達できる

#### Acceptance Criteria

1. When オーナーが候補一覧から店舗を選択したとき, the オンボーディングサービス shall 選択された店舗の店名・住所を確認できる形で提示し、確定の意思確認を行う
2. When オーナーが確定操作を行ったとき, the オンボーディングサービス shall 当該店舗を Google 上の一意な店舗（Place）と紐付けて登録し、オーナーの状態を「店舗特定済み」に遷移させる
3. When 店舗特定が完了したとき, the オンボーディングサービス shall 完了した旨と、機能1 が利用可能になる旨を案内する
4. If 選択された店舗が既に他のオーナーの店舗として登録済みのとき, the オンボーディングサービス shall 確定を行わず、その旨と運営への問い合わせ方法を案内する
5. When 確認段階でオーナーが確定を取りやめたとき, the オンボーディングサービス shall 店名の入力からやり直せる状態に戻す
6. While オーナーが「店舗特定済み」の状態, the オンボーディングサービス shall 以後のオンボーディング入力を要求しない

### Requirement 5: 会話の継続性と重複防止

**Objective:** As a 飲食店オーナー, I want 中断してもいつでも続きから再開できること, so that 忙しい業務の合間でも登録を完了できる

#### Acceptance Criteria

1. The オンボーディングサービス shall オーナーごとにオンボーディングの進捗状態を保持する
2. When オーナーが中断後に再度メッセージを送信したとき, the オンボーディングサービス shall 保存済みの進捗に応じた次の手順を案内する
3. While オンボーディング進行中, when 現在の段階で期待されない入力（テキスト以外の送信・無関係な操作等）を受信したとき, the オンボーディングサービス shall 現在の段階で必要な操作の案内を再送する
4. If 同一の操作イベントが重複して届いたとき, the オンボーディングサービス shall 登録・状態遷移・応答送信を二重に行わない
5. When オーナーがメッセージ送信または選択操作を行ったとき, the オンボーディングサービス shall 5 秒以内に応答メッセージを返す

### Requirement 6: 常設メニュー（リッチメニュー）最小形

**Objective:** As a 飲食店オーナー, I want トーク画面に常設の導線があること, so that 操作に迷ってもオンボーディングへ戻れる

#### Acceptance Criteria

1. The オンボーディングサービス shall 友だち追加後のオーナーに、オンボーディング再開の導線を含む常設メニューを表示する
2. When オンボーディング未完了のオーナーが常設メニューから再開操作を行ったとき, the オンボーディングサービス shall 現在の進捗に応じた次の手順を案内する
3. When オーナーが「店舗特定済み」へ遷移したとき, the オンボーディングサービス shall 常設メニューを完了後の案内（機能1 の利用案内を含む最小形）に切り替える

### Requirement 7: セキュリティ・プライバシー・コンプライアンス

**Objective:** As a 運営, I want 不正アクセスと規約違反が構造的に防がれること, so that テナント全体（全代理店・全オーナー）を危険に晒さない

#### Acceptance Criteria

1. The オンボーディングサービス shall LINE プラットフォーム以外を送信元とする受信リクエストを処理しない
2. The オンボーディングサービス shall オーナーについて LINE 上の識別子・表示名以外の個人情報を収集しない
3. The オンボーディングサービス shall 来店客に関する情報を一切扱わない
4. The オンボーディングサービス shall オーナーへの案内文をすべて日本語で提供する
5. If オーナーの操作を処理できない内部障害が発生したとき, the オンボーディングサービス shall 応答のないまま放置せず、再試行の案内または運営への問い合わせ方法を提示する
