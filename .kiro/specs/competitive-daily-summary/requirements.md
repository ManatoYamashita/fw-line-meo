# Requirements Document

## Project Description (Input)

**Source**: GitHub Issue #4 (https://github.com/ManatoYamashita/fw-line-meo/issues/4) ／ requirements.md §3.3

**(a) 誰が抱える課題か**
飲食店オーナー（IT に不慣れ、日次で自店の市場ポジションを把握する手段を持たない）。

**(b) 現状**
オーナーは自店が近隣の同カテゴリ競合と比べてどのような評価・順位にあるかを知る手段がなく、LINE 以外のツール（Google ダッシュボード等）を使いこなす前提のサービスは定着しない。現時点でこの機能は未実装（実装コード未着手）。

**(c) 何を変えるべきか**
毎朝、Go バッチが Cloud Scheduler 起動で全店舗 × 競合5店の評価・クチコミ数・新着クチコミを Google Places API から並行取得し、順位・前日比を算出したうえで、結論ファーストの LINE Flex Message を各店へ配信する。オーナーは LINE を開くだけで自店の市場ポジションを把握できるようにする。

### スコープ（Issue #4 より）
- 競合自動抽出（半径1km・同カテゴリ・上位5店、システムが自動抽出）
- Places API による自店＋競合の評価・クチコミ数・新着クチコミ取得（Go による並行バッチ）
- コスト最適化（競合リストは固定し指標のみ日次更新、約6回/日/店に抑制）
- 評価・順位の時系列保存と前日比算出
- Flex Message 生成（結論ファースト：順位＋前日比、星・クチコミ数、競合一覧）
- 配信時刻：デフォルト7:00・オーナー設定可
- アクション「詳細を見る」（LIFF で自店・競合の詳細を閲覧のみ表示。書込権限＝OAuth 不要、Places API の範囲で完結）

### 完了条件（Issue #4 より）
Cloud Scheduler で日次起動し、対象店舗へ Flex Message が配信される。

## Introduction

本機能(機能1: 競合ポジショニング日次サマリー)は、店舗特定を完了した飲食店オーナーに対し、毎朝 LINE Flex Message で「自店が近隣の同カテゴリ競合の中でどのポジションにいるか」を結論ファーストで届ける MVP 機能である。システムは店舗特定時に競合5店を自動抽出して固定し、以後は自店＋競合の星評価・クチコミ総数を日次で取得・時系列保存し、順位と前日比を算出して配信する。データ源は Google Places API のみとする(プロダクト不可侵制約: スクレイピング禁止)。

**要件確定の根拠**: GitHub Issue #4 ／ ルート `requirements.md` §3.3・§4・§6・§8 ／ ユーザー確認事項(2026-07-11): 順位基準＝星評価降順・同率はクチコミ総数降順、配信対象＝店舗特定済み全員(停止機能なし)、競合リスト＝店舗特定時に自動抽出し以後固定、新着クチコミ＝自店分のみ表示。

## Boundary Context

- **In scope**: 競合の自動抽出と固定／自店＋競合の指標の日次取得・時系列保存／順位・前日比の算出／LINE Flex Message の日次配信(配信時刻のオーナー設定を含む)／「詳細を見る」閲覧画面。
- **Out of scope**(第2フェーズ以降): 配信停止(オプトアウト)設定／競合範囲・店数・リストのオーナー調整／クチコミ返信・Google 投稿作成(OAuth 連携全般)／詳細分析画面／収益化。
- **Adjacent expectations**:
  - オーナーの LINE 友だち登録・店舗特定(Place 確定)までのオンボーディングは **LINE 基盤(Issue #6)** の責務であり、本機能はその完了状態を配信開始の前提として参照する(本機能は所有しない)。
  - 4階層データモデルおよび競合・評価時系列の保存構造は **`four-tier-data-model` spec** で確立済みであり、本機能はそれを唯一の真実として参照する。
  - 日次バッチの起動・失敗検知の基盤は **`gcp-infra-foundation` spec** で構築済みであり、本機能はその上で動作する。

## Requirements

### Requirement 1: 競合の自動抽出と固定

**Objective:** As a 飲食店オーナー, I want 自店の競合となる近隣同カテゴリ店をシステムが自動で選んでくれること, so that IT 知識がなくても比較対象を自分で調べ・設定する必要がない

#### Acceptance Criteria

1. When オーナーの店舗特定(Place 確定)が完了した, the system shall 自店から半径 1km 以内かつ Google 店舗カテゴリが自店と一致する店舗のうち、自店に近い順に最大 5 店を競合として自動抽出し、競合リストとして固定する
2. If 条件を満たす店舗が 5 店未満しか見つからない, the system shall 見つかった店舗のみで競合リストを構成する
3. If 条件を満たす店舗が 1 店も見つからない, the system shall 競合なしの状態で自店指標のみの日次サマリー配信を行い、メッセージ内で競合が見つかっていない旨を表示する
4. The system shall 固定した競合リストの再抽出・追加・削除の手段を MVP では提供しない(オーナーによる調整は第2フェーズ)
5. If 固定済みの競合店舗が閉店等により指標を取得できなくなった, the system shall 当該店舗を以後の順位算出・配信の比較対象から除外し、過去の時系列記録は保持する

### Requirement 2: 指標の日次取得と時系列保存

**Objective:** As a 飲食店オーナー, I want 自店と競合の評価指標が毎日自動で記録されること, so that 順位や前日比といった「変化」を毎朝把握できる

#### Acceptance Criteria

1. The system shall 店舗特定が完了した全店舗について、自店および固定済み競合の星評価・クチコミ総数を Google Places API から 1 日 1 回取得する
2. The system shall 競合データの取得を Google Places API のみで行い、スクレイピング等の規約外手段を用いない
3. When 日次取得が完了した, the system shall 取得した星評価・クチコミ総数を店舗ごと・日付ごとの時系列として保存する
4. When 日次取得が完了した, the system shall 自店＋当日有効な競合の集合内で、星評価の降順(同率の場合はクチコミ総数の降順)により自店の順位を算出し記録する
5. If 一部店舗の指標取得に失敗した, the system shall 取得に成功した店舗の処理を継続し、失敗した店舗と理由を運用者が検知できるよう記録する
6. When 同一日に日次処理が再実行された, the system shall 同日分の記録を重複させない
7. The system shall 外部 API の呼び出しを 1 店舗あたり約 6 回/日(自店 1 ＋競合最大 5)に抑える(競合リスト固定・指標のみ日次更新によるコスト最適化)

### Requirement 3: 日次サマリーの LINE 配信

**Objective:** As a 飲食店オーナー, I want 毎朝決まった時刻に LINE で自店の市場ポジションが届くこと, so that アプリや Web 管理画面を開かなくても LINE を見るだけで自店の状況がわかる

#### Acceptance Criteria

1. The system shall 店舗特定が完了した全オーナーに対し、日次サマリーを LINE Flex Message(カード型)で毎日 1 回配信する
2. The system shall 配信時刻のデフォルトを朝 7:00 とする
3. When オーナーが LINE 上の設定で配信時刻を変更した, the system shall 翌日以降の配信に変更後の時刻を適用する
4. The system shall 配信メッセージを結論ファーストの優先順位で構成する: (1) 自店の順位(近隣 N 店中 △位)と前日比(上昇/下降/変動なし)、(2) 自店の星評価・クチコミ総数、(3) 自店の新着クチコミ、(4) 競合との星差・競合一覧(簡潔に)
5. When 自店のクチコミ総数が前日より増加している, the system shall 新着件数と、取得可能な場合はその内容をメッセージに含める(表示は自店の新着のみとし、競合の新着は表示しない)
6. If 自店に新着クチコミがない, the system shall 新着クチコミ欄を「新着なし」として表示する
7. If 前日の記録が存在しない(初回配信等), the system shall 前日比を表示せず、当日の順位・指標のみで配信する
8. If 特定オーナーへの配信が失敗した, the system shall 他オーナーへの配信を継続し、失敗を運用者が検知できるよう記録する
9. When 同一日に配信処理が再実行された, the system shall 同一オーナーへ同日のサマリーを重複配信しない
10. The system shall 配信停止(オプトアウト)手段を MVP では提供しない
11. The system shall メッセージを日本語で提供する

### Requirement 4: 「詳細を見る」閲覧画面

**Objective:** As a 飲食店オーナー, I want サマリーから 1 タップで自店・競合の詳細を確認できること, so that 気になった変化の背景を LINE から離れずに確認できる

#### Acceptance Criteria

1. When オーナーが日次サマリーの「詳細を見る」アクションをタップした, the system shall LINE 内で自店および競合の詳細(星評価・クチコミ総数・順位の推移を含む)を閲覧画面として表示する
2. The system shall 詳細画面を閲覧のみとし、クチコミ返信・投稿作成等の書込操作を一切提供しない(Google アカウントの書込権限・OAuth 認可を要求しない)
3. While オーナーの店舗で競合が 1 店も固定されていない, the system shall 詳細画面で自店の指標のみを表示する
4. The system shall 詳細画面を日本語で提供する

### Requirement 5: 運用可観測性

**Objective:** As a 運営者, I want 日次処理と配信の成否が検知できること, so that オーナーが「今朝サマリーが届かない」と気づく前に異常へ対処できる

#### Acceptance Criteria

1. If 日次処理全体が起動しなかった、または異常終了した, the system shall 運用者が当日中に検知できるよう失敗を記録・通知可能にする
2. The system shall 日次処理の実行ごとに、対象店舗数・取得成功/失敗数・配信成功/失敗数を運用者が確認できるよう記録する
