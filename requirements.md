# 要件定義書：飲食店向け LINE 一元管理アプリ（仮称：LINE Restaurant Manager）

- 文書バージョン: 1.0
- 作成日: 2026-06-25
- ステータス: ドラフト（要件定義フェーズ完了）

---

## 1. 背景・目的

### 1.1 解決したい課題
飲食店オーナーは多忙で、IT リテラシーが高くない傾向にある。一方で **LINE だけは日常的に使える**。
従来の複雑な Web 管理ツールはオーナーにとって学習・運用コストが高く、定着しない。

### 1.2 コンセプト
**「飲食店向けに複雑な Web UI を持たず、LINE で全てを一元管理できるアプリ」**

- オーナーが触れる窓口は原則 **LINE（チャット + リッチメニュー、必要箇所のみ LIFF）** に限定する。
- 「複雑な Web UI を持たない」という思想は **オーナー向け** に適用される方針であり、
  オーナー以外のアクター（代理店・運営）には Web ダッシュボードを提供する。

### 1.3 提供価値（要約）
飲食店オーナーが、ITに不慣れでも LINE だけで
「自店の市場ポジション把握」「Google クチコミの獲得促進」「Google ビジネス投稿（将来）」
を完結できるようにする。

---

## 2. システム全体像

### 2.1 プラットフォーム構成
| 区分 | 採用技術 | 用途 |
|---|---|---|
| 主軸 | LINE Messaging API（公式アカウント／Bot） | オーナーへのプッシュ配信・チャット操作・リッチメニュー |
| 補助 | LIFF（LINE Front-end Framework） | LINE 内で完結する軽い入力（店舗特定など） |
| 客向け | 通常 Web ページ（LIFF ではない） | 機能3 のアンケート（来店客が利用、LINE 不要） |

> MINI App は審査が重く初期不要のため採用しない。

### 2.2 テナント構成
- **マルチテナント型**：運営が保有する **単一の LINE 公式アカウント** に全オーナーが友だち登録する。
- 客向け機能（機能3）は LINE を経由しない（客をLINEに登録させない）。

### 2.3 登場アクター（4階層・確定）
```
運営（開発者／我々）
   └─ 代理店（Agency）
         └─ 飲食店オーナー（Owner）
               └─ 来店客（Customer）※匿名
```
- **運営と代理店は「チーム」として同一の Web ダッシュボードにログイン**し、ロール（RBAC）で権限を分離する。
  - 運営: 全代理店・全店舗を閲覧可能（管理者）
  - 代理店: 自身の担当店のみ閲覧・操作可能
- データモデルはこの4階層を **初期から確定構造** として設計する（後からの階層挿入は不可）。

## 3. 機能要件

### 3.1 機能一覧と優先順位
| No | 機能 | 外部依存 | 難易度 | MVP | 提供フェーズ |
|---|---|---|---|---|---|
| 機能3 | 口コミ用 QR・アンケート | ほぼ無し | 低 | ◎ | MVP（最優先） |
| 機能1 | 競合ポジショニング日次サマリー | Google Places API | 高 | ◎ | MVP |
| 機能2 | Google ビジネスプロフィール投稿作成 | GBP API（OAuth・審査） | 中〜高 | × | 第2フェーズ |
| 機能1-b | クチコミ返信（自店レビューへの返信） | GBP API（OAuth・書込） | 中 | × | 第2フェーズ |

> 機能2 と機能1-b は **同一の Google OAuth 基盤を共有** するため、OAuth 連携を一度構築すれば同時に解禁できる。

---

### 3.2 機能3：口コミ用 QR・アンケート（MVP 最優先）

#### 3.2.1 前提（Google 規約）
- Google クチコミは **客本人が Google アカウントで投稿** する必要がある（API 代理投稿は規約違反）。
- 本システムの責務は **「下書き生成 → 客がコピペして Google に貼る」** までとする。
- **レビューゲーティング（高評価のみ Google 誘導／低評価を隠す）は Google 規約違反のため禁止。**
  低評価の客も同一の導線で Google 投稿へ誘導する。

#### 3.2.2 利用フロー
1. **QR 設置場所**：テーブル。
2. **遷移先**：通常 Web ページ（スマホブラウザで即開く。LINE 不要・摩擦ゼロ）。
3. **アンケート**：3〜4 問・全てタップ式中心。
   - 料理の満足度（星）
   - 良かった点（複数選択：味／量／接客／雰囲気 等）
   - 一言（任意・自由記述）
4. **下書き生成**：**Gemini API** によるAI生成。
   - 客が選んだ要素・自由記述の断片を「素材」として渡す。
   - 客ごとに異なる切り口・語彙で生成し、テンプレ感（スパム検知リスク）を回避。
   - **「客本人が選んだ事実のみを書く」** 制約で、嘘・誇張を排除。
5. **投稿導線**：「コピー → Google のクチコミ欄を開く」ボタン（Place ID からクチコミ投稿 URL へ直行）。
6. **低評価対応**：そのまま Google 誘導（ゲーティングをしない）。

#### 3.2.3 AI ガードレール要件（必須）
- 嘘の体験・実在しない事実を生成しない。
- 過剰な誇張をしない。
- 公序良俗に反する内容を生成しない。
- **過剰な悪口・誹謗中傷を生成しない**（低評価でも節度ある表現に留める）。

#### 3.2.4 データ・プライバシー（重要）
- 客の氏名・連絡先など **個人情報は一切取得しない**。
- アンケート回答は **Place 単位の匿名集計のみ** 保持する（個別回答の永続保存は行わない方針）。
- 集計結果（例：今月★4以上が N 件 等）はオーナーへの付加価値として提示し得る。

---

### 3.3 機能1：競合ポジショニング日次サマリー（MVP）

#### 3.3.1 提供価値
「自店が今どれくらいの状況にいるか（市場ポジション）」を毎朝 LINE で可視化する。

#### 3.3.2 取得データ・データ源
- データ源：**Google Places API**（従量課金を許容）。
- スクレイピング・規約違反取得は行わない。
- 取得指標：自店および競合の **星評価・クチコミ総数・直近の新着クチコミ**。

#### 3.3.3 競合の定義
| 項目 | MVP 仕様 | 将来 |
|---|---|---|
| 範囲 | 自店から半径 **1km 以内** | 設定可能 |
| カテゴリ | Google 店舗カテゴリが **一致**（例：ラーメン店） | - |
| 店数 | デフォルト **5 店** | 設定可能 |
| 選定方法 | **システムが自動抽出** | オーナーがライバル店リストを追加・調整可能 |

#### 3.3.4 配信仕様
- **配信時刻**：デフォルト **朝 7:00**、設定からオーナーが変更可能。
- **形式**：**LINE Flex Message**（カード型・視覚表現）。
- **情報の優先順位（結論ファースト）**：
  1. 自店の順位（近隣 N 店中 △位）＋前日比（↑↓）
  2. 自店の星評価・クチコミ総数
  3. 新着クチコミ
  4. 競合との星差・競合一覧（簡潔に）
- **アクション**：
  - MVP：「詳細を見る」（LIFF で自店・競合の詳細を **閲覧のみ** 表示。書込権限＝OAuth 不要、Places API の範囲で完結）
  - 第2フェーズ：「クチコミに返信」「Google 投稿作成」

#### 3.3.5 コスト最適化方針
- 競合リストは固定し、評価・クチコミ数等の指標のみ日次更新する。
- API 呼び出しを **自店＋競合5店＝約6回/日/店** 程度に抑制する。

---

### 3.4 機能2：Google ビジネスプロフィール投稿作成（第2フェーズ）

- **Google Business Profile API** による OAuth 連携が必須（利用審査・制約あり ＝ 本プロジェクト最大の技術的関門）。
- **段階的連携**：オーナーはまず店舗特定のみで開始（機能1 を先行体験）。
  Google 投稿を使いたくなった時点で初めて OAuth 認可へ誘導する。
- 投稿文は Gemini で下書き生成 → オーナーが LINE 上で確認・承認 → 投稿。
- 同一 OAuth 基盤で「クチコミ返信」（機能1-b）も同時解禁する。

---

### 3.5 代理店ダッシュボード（MVP・最小形）

- 利用者：運営・代理店（同一ダッシュボード、RBAC で権限分離）。
- 認証：**Google OAuth / Identity Platform（Firebase Authentication）**。パスワードは自前管理しない。
- MVP スコープ（最小）：
  1. ログイン（運営／代理店ロール）
  2. **担当店の登録**（店名検索 → Place 特定 → 基本情報入力。＝オーナーのオンボーディングを代行）
  3. **担当店の一覧表示**（登録済み／競合設定済み 等のステータス確認）
- 第2フェーズ以降の贅肉（店ごとの詳細分析画面・売上レポート等）は MVP に含めない。

---

## 4. オンボーディング設計

- 方針：**段階的連携**。
  1. 入口は **店舗の特定のみ**（代理店が代行：店名検索 → Place 選択。店の身元＝Place ID・場所・カテゴリを確定）。
     → この時点で **機能1 が即稼働** し、価値を先行体験させる。
  2. システムが「半径1km・同カテゴリ・クチコミ数上位5店」を自動提案 → オーナー/代理店が微調整。
  3. Google 投稿（機能2）を使う段階で初めて Google OAuth 認可へ誘導（第2フェーズ）。

## 5. 技術アーキテクチャ

### 5.1 クラウド基盤：GCP で統一
（Places API・GBP・Gemini への依存が Google に集中するため、認証・課金・運用を単一帝国に統合）

| レイヤ | 採用技術 | 役割 |
|---|---|---|
| 実行環境 | **Cloud Run** | LINE Webhook・客向け Web・ダッシュボード API（リクエスト時起動・ゼロスケール） |
| スケジューラ | **Cloud Scheduler + Cloud Run** | 日次バッチの確実な起動 |
| データベース | **Cloud SQL（PostgreSQL）** | 店舗・競合・評価/順位の時系列・（将来）OAuth トークン |
| 認証（ダッシュボード） | **Identity Platform / Firebase Auth（Google ログイン）** | 運営・代理店のログイン、ロール管理 |
| 生成AI | **Gemini API** | 口コミ下書き・（将来）Google 投稿/返信文の生成 |

### 5.2 言語・処理分担（二刀流）
| 言語 | 担当層 | 理由 |
|---|---|---|
| **TypeScript** | リアルタイム応答層 | LINE Webhook・リッチメニュー・Flex Message 組立（LINE 公式 SDK が充実）／客向けアンケート Web（型共有）／Gemini 呼び出しのオーケストレーション |
| **Go** | 日次バッチ層 | 全店舗 × 競合5店の Places API を毎朝並行取得。goroutine による高速・低メモリ・安定処理。Cloud Run コスト削減 |

#### 二刀流の運用上の注意
- DB を 2 言語から触るため、**書き込み境界（どの言語がどのテーブルを書くか）を厳格に定義** する。
- 共有定数（カテゴリ定義等）の同期に注意する。

## 6. 非機能要件

| 区分 | 要件 |
|---|---|
| セキュリティ | ダッシュボードは Google ログイン（Identity Platform）。客の個人情報は取得しない（持たないデータは漏れない） |
| プライバシー | アンケート回答は Place 単位の匿名集計のみ。氏名・連絡先を取得しない |
| 規約遵守 | レビューゲーティング禁止／Google クチコミは客本人投稿／スクレイピング禁止 |
| コスト | 競合リスト固定・指標のみ日次更新で Places API 呼び出しを最小化 |
| スケーラビリティ | Cloud Run のゼロスケールで初期コストを抑制、店舗増に追従 |
| 言語 | 日本語（UI・生成文ともに） |

## 9. 主要な技術的リスク・論点

| リスク | 内容 | 対応方針 |
|---|---|---|
| GBP API 審査 | Google Business Profile API は利用審査・制約があり、承認に時間を要する／降りない可能性 | MVP から除外し第2フェーズへ。MVP の生死を Google の審査に賭けない |
| クチコミスパム検知 | 類似下書きの量産は Google にスパム判定され、最悪ビジネスプロフィールごと停止 | 客ごとに切り口・語彙を変えて生成、客本人が選んだ事実のみ反映 |
| Places API コスト | 店舗数 × 競合数 × 日次で従量課金が増大 | 競合固定・指標のみ日次更新で呼び出し最小化 |
| 二刀流の複雑性 | TS/Go の 2 系統で運用・型管理が二重化 | 書込境界の厳格定義、共有定数の同期管理 |
